# Change Workflow

本文件把「先写文档，再改代码」落实为可执行流程。

## 1. 标准流程

每次要修改仓库时，按照以下顺序执行：

1. 判断变更类型
2. 找到必须先更新的文档
3. 更新文档
4. 再进入实现
5. 完成后同步摘要文档
6. 自查是否留下文档断链、重复说明或规则冲突

## 2. 变更类型决策表

| 变更类型 | 先更新这些文档 |
| --- | --- |
| 改当前阶段、目标、允许范围、禁止事项 | `docs/spec.md`、`docs/guide.md` |
| 改首页区块文案、布局、阅读顺序 | `docs/architecture.md`、`docs/homepage-component-contracts.md`、`README.md` |
| 改首页组件职责或状态边界 | `docs/architecture.md`、`docs/homepage-component-contracts.md` |
| 改数据字段、品种枚举、资料目录 | `docs/data-model.md`、`README.md` |
| 改脚本命令、运行方式、验证流程 | `docs/development.md`、`README.md` |
| 改文档规则、协作流程、门禁 | `docs/guide.md`、`docs/documentation-structure.md`、`AGENTS.md` |
| 新增专题文档 | `docs/guide.md`、`docs/documentation-structure.md` |

## 3. 完成后同步规则

实现完成后，至少检查以下事项：

- `README.md` 是否仍然只是摘要入口，而没有复制权威事实
- `AGENTS.md` 是否仍然只保留门禁，而没有混入细节规范
- `docs/guide.md` 是否能找到这次涉及的专题文档
- 是否新增了专题文档却没有挂到 `docs/guide.md`
- 是否把本该只存在于一个权威文档中的内容复制到了第二个文件

## 4. 文档更新模板

进入实现前，至少补清楚以下项目：

| 变更项 | 需要写清楚的问题 |
| --- | --- |
| 当前阶段 | 目标是否改变、哪些内容进入或离开范围 |
| 页面结构 | 改了哪些区块、职责是否改变 |
| 状态流 | 新状态由谁维护、谁消费 |
| 数据结构 | 字段是否新增/删除/改名，允许值从哪里来 |
| 脚本流程 | 命令有无变化、执行顺序是否变化 |
| 协作规则 | 是否影响“先写文档再改代码”的门禁 |

## 5. 自查清单

提交前快速检查：

- 是否先改了文档，再改实现
- 是否有两份文档在描述同一规则但表述冲突
- 是否已经确认这次变更的“权威文档”只有一个
- 是否把详细规则错误写进了 `AGENTS.md`
- 是否把摘要性内容过度塞回 `README.md`
- 是否新增了未被入口文档收录的文档
