# 代码维护约定

## 注释原则

- 注释的目标是帮助后来维护者快速理解背景、约束与扩充边界。
- 优先为下列情境补充注释：
  - 资料是如何推导或生成的。
  - 规则看似随意、其实是当前阶段约束的一部分。
  - 型别或资料结构需要与其他档案保持同步。
  - 某段逻辑未来很可能被替换、扩写或接上真实资料来源。
- 避免撰写逐行翻译式注释，例如把明显的赋值、回传或条件判断再写成自然语言。

## 推荐位置

- 档案开头：说明该档案在当前阶段的角色。
- 复杂常数或映射：说明来源、用途或与资料契约的关系。
- 非直觉计算：说明为什么这样算，以及哪些地方不能随意改动。
- 对外暴露的型别：说明其与资料脚本或 UI 的耦合点。

## 同步要求

- 新增功能或重构时，若原有注释已失真，必须同步更新或移除。
- 变更脚本、型别或资料契约时，要检查注释是否仍与 `docs/spec.md`、`docs/data-model.md` 一致。
