# Documentation Structure

本文件说明 `docs/` 目录中每份文件的职责边界，避免文档重复或互相矛盾。

## 1. 设计目标

文档结构要解决三件事：

- 新协作者能快速找到入口
- 变更时知道该更新哪份文件
- 避免所有规则都堆在 `README.md`

## 2. 文件职责

| 文件 | 主要职责 | 什么时候必须更新 |
| --- | --- | --- |
| `docs/guide.md` | 文档入口、查阅路径、维护规则 | 文档入口、协作流程改变时 |
| `docs/spec.md` | 当前阶段、目标、允许范围、禁止事项 | 阶段目标、in/out of scope 改变时 |
| `docs/architecture.md` | 首页实现规范、模块职责、状态流 | 组件职责、目录分层、页面结构改变时 |
| `docs/homepage-component-contracts.md` | 首页核心组件契约与修改边界 | 首页组件职责、输入依赖、非职责改变时 |
| `docs/data-model.md` | 资料来源、字段结构、允许值来源 | 数据字段、产物目录、脚本输入输出改变时 |
| `docs/development.md` | 安装、命令、开发工作流 | 启动方式、脚本命令、验证流程改变时 |
| `docs/change-workflow.md` | 文档优先的变更流程与决策表 | 协作工作流、同步规则改变时 |
| `README.md` | 对外摘要与快速开始 | 任一高层资讯变动时同步摘要 |
| `AGENTS.md` | 协作门禁与强制规则 | 协作规则改变时 |

## 3. 推荐原则

- 入口统一从 `docs/guide.md` 开始。
- 详细说明写入专题文档，不在 `README.md` 复制整段长说明。
- `README.md` 负责让第一次进入仓库的人快速理解项目。
- `AGENTS.md` 负责约束协作行为，不承载完整专题内容。

## 4. 命名规则

新增专题文档时，命名应遵守以下规则：

- 优先使用直白的功能语义命名，例如 `change-workflow.md`
- 避免过度抽象或团队外难以理解的缩写
- 一个文件只负责一个稳定主题
- 新文档建立后，必须同步更新 `docs/guide.md`

## 5. 不要写进 `AGENTS.md` 的内容

以下内容不应长期写入 `AGENTS.md`：

- 首页组件逐一职责说明
- 详细字段表
- 长篇开发流程说明
- 文档命名细则
- 未来扩充设计讨论

这些内容应分别写入 `docs/architecture.md`、`docs/homepage-component-contracts.md`、`docs/data-model.md`、`docs/change-workflow.md`。
当前阶段与范围应写入 `docs/spec.md`。

## 6. 更新策略

### 小改动

若只是补充说明、修正文案或更新路径，可直接改对应专题文件，并检查 `README.md` 是否需要同步摘要。

### 结构性改动

若调整目录、页面结构、数据结构或开发流程，建议顺序为：

1. 更新 `docs/guide.md` 中的入口描述或规则（如需要）
2. 更新对应专题文件
3. 同步 `README.md`
4. 若协作规则被影响，再更新 `AGENTS.md`

## 7. 后续扩充建议

未来若新增：

- API 串接
- 领养流程
- 设计规范
- 测试策略

可继续在 `docs/` 下新增专题文件，但需要先在 `docs/guide.md` 与本文件补上入口与职责说明。
